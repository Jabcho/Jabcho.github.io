# Directory Structure
<br>
Jekyll 사이트의 기본 구조는 다음과 같다:

```
.
├── _config.yml
├── _data
│   └── members.yml
├── _drafts
│   ├── begin-with-the-crazy-ideas.md
│   └── on-simplicity-in-technology.md
├── _includes
│   ├── footer.html
│   └── header.html
├── _layouts
│   ├── default.html
│   └── post.html
├── _posts
│   ├── 2007-10-29-why-every-programmer-should-play-nethack.md
│   └── 2009-04-26-barcamp-boston-4-roundup.md
├── _sass
│   ├── _base.scss
│   └── _layout.scss
├── _site
├── .jekyll-cache
│   └── Jekyll
│       └── Cache
│           └── [...]
├── .jekyll-metadata
└── index.html # can also be an 'index.md' with valid front matter
```

<div class="yellow-caution" markdown="1">
✨ **gem 기반 테마를 사용하는 Jekyll 사이트의 디렉터리 구조**

Jekyll <code class="code-inline">3.2</code>부터 <code class="code-inline">jekyll new</code>로 생성한 새 프로젝트는 [gem 기반 테마](#themes)를 사용하여 사이트의 레이아웃과 스타일을 구성한다. 이렇게 하면 기본적으로 <code class="code-inline">_layouts</code>, <code class="code-inline">_includes</code>, <code class="code-inline">_sass</code>가 테마 gem(= theme-gem) 안에 포함되기 때문에 기본 디렉터리 구조가 가벼워진다.

현재 기본 테마는 [minima](https://github.com/jekyll/minima)이며, <code class="code-inline">bundle info minima</code>를 실행하면 로컬 PC에서 minima 테마 파일이 저장된 위치를 확인할 수 있다.
</div>

```yaml
include:
  - _pages
  - .htaccess
```

# Liquid
<br>
Jekyll은 템플릿을 처리하기 위해 [Liquid](https://shopify.github.io/liquid/) 템플릿 언어를 사용한다.

일반적으로 Liquid에서 콘텐츠 출력은 중괄호 2개(예: {% raw %}{{ variable }}{% endraw %}), 제어문은 중괄호 한 개와 퍼센트 기호로 감싼 구문(예: {% raw %}{% if statement %}{% endraw %})을 사용한다. Liquid에 대한 더 자세한 내용은 [공식 Liquid 문서](https://shopify.github.io/liquid/)를 참고하라.

Jekyll은 사이트를 구축할 때 유용한 여러 가지 Liquid 확장 기능을 제공한다:

- [필터(Filters)]({% link _apis/jekyll/docs/01/liquid_filters.md %})

- [태그(Tags)]({% link _apis/jekyll/docs/01/tags_filters.md %})

# Variables
<br>
Jekyll은 사이트를 순회하며 처리할 파일을 찾는다. Front matter가 있는 파일은 모두 처리 대상이다. 각 파일마다 Jekyll은 Liquid에서 사용할 수 있는 다양한 변수 및 속성을 제공한다. 다음은 사용 가능한 변수 목록을 표로 정리한 것이다.

✨ **ProTip™: 사용자 지정 Front matter 사용**
Front matter에 지정한 사용자 정의 항목은 <code class="code-inline">page</code>로 접근할 수 있다. 예를 들어, 페이지의 Front matter에 <code class="code-inline">custom_css: true</code>를 지정하면 그 값(true)은 <code class="code-inline">page.custom_css</code>로 사용할 수 있다.
레이아웃의 Front matter는 <code class="code-inline">layout</code>을 통해 접근할 수 있다. 예를 들어, 레이아웃의 Front matter에 <code class="code-inline">class: full_page</code>를 지정하면 해당 값(full_page)은 해당 레이아웃 및 그 상위 레이아웃에서 <code class="code-inline">layout.class</code>로 사용할 수 있다.

# Includes
<br>
<code class="code-inline">include</code> 태그를 이용하면 <code class="code-inline">_includes</code> 디렉터리에 있는 다른 파일의 내용을 현재 문서에 삽입할 수 있다.

```liquid
{% raw %}{% include footer.html %}{% endraw %}
```

Jekyll은 소스 디렉터리 루트에 있는 <code class="code-inline">_includes</code>에서 참조 파일(위 예시의 경우 <code class="code-inline">footer.html</code>)을 찾아서 그 내용을 현재 위치에 삽입한다.

## Including files relative to another file
<code class="code-inline">include_relative</code> 태그를 사용하면 현재 파일 기준의 상대 경로에 있는 조각 파일을 불러올 수 있다. 

```liquid
{% raw %}{% include_relative somedir/footer.html %}{% endraw %}
```

대상 파일을 반드시 <code class="code-inline">_includes</code> 디렉터리에 둘 필요는 없다. 대신 태그가 사용된 파일의 위치를 기준으로 상대 경로를 해석한다. 예를 들어 <code class="code-inline">_posts/2014-09-03-my-file.markdown</code>에서 <code class="code-inline">include_relative</code>를 사용하면, 포함되는 파일은 반드시 <code class="code-inline">_posts</code> 또는 그 하위 디렉터리 안에 있어야 한다.

<code class="code-inline">../</code> 문법을 사용한 상위 디렉터리를 참조는 불가능하다는 점에 유의하라.

그 외 변수 전달 등 <code class="code-inline">include</code> 태그가 제공하는 기능은 <code class="code-inline">include_relative</code>에서도 동일하게 사용할 수 있다.

## Using variables names for the include file
삽입하려는 파일의 이름은 실제 파일명을 쓰는 대신 변수로 지정할 수 있다. 예를 들어, 페이지의 Front matter에 다음과 같이 변수를 정의했다고 하자:

```yaml
---
title: My page
my_variable: footer_company_a.html
---
```

이제 include에서 해당 변수를 참조할 수 있다:

```liquid
{% raw %}{% if page.my_variable %}
  {% include {{ page.my_variable }} %}
{% endif %}{% endraw %}
```

위 예에서 include는 <code class="code-inline">_includes/footer_company_a.html</code> 디렉터리에 있는 <code class="code-inline">footer_company_a.html</code> 파일을 삽입한다.

## Passing parameters to includes
include에 매개변수를 전달할 수 있다. 예를 들어, <code class="code-inline">_includes</code> 폴더에 다음과 같은 형식의 <code class="code-inline">note.html</code> 파일이 있다고 하자:

```liquid
<div markdown="span" class="alert alert-info" role="alert">
  <i class="fa fa-info-circle"></i> <b>Note:</b>
  {% raw %}{{ include.content }}{% endraw %}
</div>
```

여기서 <code class="code-inline">{% raw %}{{ include.content }}{% endraw %}</code>는 include를 호출할 때 해당 매개변수에 값을 지정하면 그 값으로 치환되는 매개변수로, 다음과 같이 사용할 수 있다:

```liquid
{% raw %}{% include note.html content="This is my sample note." %}{% endraw %}
```

이 경우 <code class="code-inline">content</code>의 값(즉, <code class="code-inline">This is my sample note</code>)이 <code class="code-inline">{% raw %}{{ include.content }}{% endraw %}</code> 위치에 삽입된다.

include에 매개변수를 전달하는 방식은 Markdown 본문에서 복잡한 서식 처리를 숨길 수 있어 특히 유용하다.

예를 들어, 복잡한 서식 처리가 필요한 특별한 이미지 문법이 있고, 작성자가 복잡한 서식을 매번 기억하길 원치 않는다고 하자. 이럴 때 include와 매개변수를 이용해 서식을 단순화할 수 있다. 다음은 include에 전달할 특별한 이미지 문법의 예시다:

```html
<figure>
   <a href="https://jekyllrb.com">
     <img src="logo.png" style="max-width: 200px;" alt="Jekyll logo" />
   </a>
   <figcaption>This is the Jekyll logo</figcaption>
</figure>
```

이 내용을 include 파일로 템플릿화(templatize)하여 각 값을 매개변수로 받을 수 있게 만들면 다음과 같다:

```liquid
<figure>
  {% raw %}<a href="{{ include.url }}">
    <img src="{{ include.file }}" style="max-width: {{ include.max-width }};" alt="{{ include.alt }}" />
  </a>
  <figcaption>{{ include.caption }}</figcaption>{% endraw %}
</figure>
```

이 include는 다음 5개의 매개변수를 가진다:

- <code class="code-inline">url</code>

- <code class="code-inline">max-width</code>

- <code class="code-inline">file</code>

- <code class="code-inline">alt</code>

- <code class="code-inline">caption</code>

아래는 이 include(파일명: image.html)에 모든 매개변수를 전달하는 예시다:

```liquid
{% raw %}{% include image.html url="https://jekyllrb.com"
   max-width="200px" file="logo.png" alt="Jekyll logo"
   caption="This is the Jekyll logo." %}{% endraw %}
```

결과는 앞서 제시한 원본 HTML과 동일하게 출력된다.

작성자가 매개변수 값을 지정하지 않은 경우에 대비하여 [Liquid의 default 필터](https://shopify.github.io/liquid/filters/default/)를 사용할 수 있다.

include는 오디오나 비디오 삽입, 알림, 특수 포맷 등 다양한 용도의 템플릿으로 활용할 수 있다. 다만 include를 과도하게 사용하면 사이트 빌드 시간이 길어질 수 있다. 가령 이미지를 삽입할 때마다 include를 쓰는 방식은 피하는 편이 좋다. (위 예시는 특수한 이미지에 대한 유스케이스이다.)

## Passing parameter variables to includes
전달하려는 매개변수가 문자열이 아니라 변수일 수도 있다. 예를 들어, 실제 제품명을 하드코딩하는 대신 <code class="code-inline">{% raw %}{{ site.product_name }}{% endraw %}</code>를 이용해 해당 제품이 쓰이는 모든 곳을 참조하도록 할 수 있다. (이 경우 <code class="code-inline">_config.yml</code>파일에 <code class="code-inline">product_name</code> 키를 두고, 값으로 제품명을 지정한다.)

이때 include로 전달하는 매개변수는 중괄호를 포함할 수 없다. 예를 들어 다음과 같은 문장은 매개변수로 전달할 수 없다: <code class="code-inline">"The latest version of {{ site.product_name }} is now available."</code>

이 변수를 포함한 매개변수를 include로 전달하려면, 먼저 매개변수 값 전체를 하나의 변수로 저장해야 한다. 이 변수는 <code class="code-inline">capture</code> 태그를 이용해 만들 수 있다:

```liquid
{% raw %}{% capture download_note %}
The latest version of {{ site.product_name }} is now available.
{% endcapture %}{% endraw %}
```

그 다음, 이 캡처한 변수를 include의 매개변수로 전달한다. 더 이상 문자열이 아닌 변수이므로, 매개변수 값 주변의 따옴표는 제거한다:

```liquid
{% raw %}{% include note.html content=download_note %}{% endraw %}
```

# Layouts
<br>
레이아웃은 콘텐츠를 감싸는 템플릿이다. 레이아웃을 사용하면 템플릿의 소스 코드를 한 곳에서 관리할 수 있어, 내비게이션이나 푸터 같은 공통 요소를 모든 페이지에 중복 작성하지 않아도 된다.

레이아웃은 <code class="code-inline">_layouts</code> 디렉터리에 위치한다. 일반적으로 <code class="code-inline">default.html</code>을 기본 레이아웃으로 두고, 나머지는 필요에 따라 이를 [상속](#inheritance)한다.

<div class="yellow-caution" markdown="1">
✨ **레이아웃 디렉터리**

Jekyll은 사이트 <code class="code-inline">source</code> 또는 테마의 루트에 있는 <code class="code-inline">_layouts</code> 디렉터리를 찾는다.
_config.yml의 layouts_dir 키로 레이아웃이 위치할 디렉터리 이름을 바꿀 수 있지만, 해당 디렉터리 자체는 사이트 <code class="code-inline">source</code> 디렉터리의 루트에 있어야 한다.
</div>

## Usage
먼저 레이아웃의 템플릿 소스 코드를 <code class="code-inline">default.html</code>에 작성한다. content는 레이아웃 전용 변수로, 레이아웃이 감싸는 게시글이나 페이지의 렌더링된 콘텐츠가 값으로 들어간다.

```liquid
{% raw %}<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <title>{{ page.title }}</title>
    <link rel="stylesheet" href="/css/style.css">
  </head>
  <body>
    <nav>
      <a href="/">Home</a>
      <a href="/blog/">Blog</a>
    </nav>
    <h1>{{ page.title }}</h1>
    <section>
      {{ content }}
    </section>
    <footer>
      &copy; to me
    </footer>
  </body>
</html>{% endraw %}
```

레이아웃 안에서는 원본 문서의 Front matter에 자유롭게 접근할 수 있다. 위 예시에서 <code class="code-inline">page.title</code>은 해당 페이지의 Front matter에 정의된 값이다.

다음으로, 각 페이지의 Front matter에서 사용할 레이아웃을 지정한다. 매 페이지마다 지정하지 않으려면 [Front matter 기본값(defaults)]({% link _apis/jekyll/docs/02/front_matter_defaults.md %})을 활용할 수 있다.

```markdown
---
title: My First Page
layout: default
---

This is the content of my page
```

위 페이지의 렌더링 결과는 다음과 같다.

```
<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <title>My First Page</title>
    <link rel="stylesheet" href="/css/style.css">
  </head>
  <body>
    <nav>
      <a href="/">Home</a>
      <a href="/blog/">Blog</a>
    </nav>
    <h1>My First Page</h1>
    <section>
      This is the content of my page
    </section>
    <footer>
      &copy; to me
    </footer>
  </body>
</html>
```

## Inheritance
레이아웃 상속은 사이트의 일부 문서에만 기존 레이아웃에 무언가를 덧붙이고 싶을 때 유용하다. 흔한 예로, 블로그 게시글에는 날짜와 작성자를 표시하되, 나머지는 기본 레이아웃과 동일하게 유지하고 싶을 수 있다.

이를 위해 새로운 레이아웃 파일을 하나 더 만들고, 상속할 레이아웃을 Front matter에 명시한다. 예를 들어, _layouts/post.html을 만들고 다음과 같이 작성한다:

```liquid
---
layout: default
---
<p>{% raw %}{{ page.date }} - Written by {{ page.author }}</p>

{{ content }}{% endraw %}
```

이제 게시글은 이 레이아웃을 사용하고, 그 밖의 페이지들은 기본 레이아웃(default)을 그대로 사용할 수 있다.

## Variables
레이아웃 파일에도 Front matter를 설정할 수 있다. 단, Liquid에서 사용할 때는 <code class="code-inline">page</code> 대신 <code class="code-inline">layout</code> 변수를 사용해야 한다. 

예:

```liquid
---
city: San Francisco
---
<p>{% raw %}{{ layout.city }}</p>

{{ content }}{% endraw %}
```

# Permalinks
<br>
Permalinks는 페이지, 게시글, 컬렉의 출력 경로이다. 이를 사용하면 소스 코드와 출력 디렉터리 구조를 다르게 설계할 수 있다. 

## Front Matter
문서의 Front matter를 이용해 간단히 permalink를 설정할 수 있다. 원하는 출력 경로를 <code class="code-inline">permalink</code> 변수에 적어주면 된다. 

예를 들어, 페이지 소스가 <code class="code-inline">/my_pages/about-me.html</code>에 있고 출력 URL을 <code class="code-inline">/about/</code>로 만들고 싶다면 해당 페이지의 Front matter에 다음과 같이 작성한다: 

```yaml
---
permalink: /about/
---
```

## Global
사이트 내 모든 페이지의 Front matter에 permalink를 설정하는 대신, <code class="code-inline">_config.yml</code>에서 전역 permalink 구조를 설정할 수 있다. 

<code class="code-inline">_config.yml</code>의 <code class="code-inline">permalink</code> 변수를 이용해 전역 permalink를 설정할 수 있으며, placeholders를 이용해 원하는 출력 형태를 지정한다. 

예: 

```yaml
permalink: /:categories/:year/:month/:day/:title:output_ext
```

게시글(<code class="code-inline">posts</code>)과 초안(<code class="code-inline">drafts</code>)를 제외한 페이지(pages)와 컬렉션에는 시간과 카테고리 요소가 없으므로, 해당 요소가 포함된 permalink 스타일을 지정해도 그 부분은 무시된다. 페이지에서 <code class="code-inline">:title</code>은 <code class="code-inline">:basename</code>과 동일하게 동작한다. 

따라서 위 예시 <code class="code-inline">/:categories/:year/:month/:day/:title:output_ext</code>를 페이지나 <code class="code-inline">posts</code>나 <code class="code-inline">drafts</code> 외 페이지나 컬렉션에 적용하면 결과는 /<code class="code-inline">:title.html</code>이 된다.

### Placeholders
다음은 사용 가능한 placeholders의 목록이다:
<style>
  /* ▼ 열 비율을 %로 조정 (합계 100%) */
  :root {
    --var-col-pct: 28%;
  }

  /* ⬇ 자동 줄바꿈 강화(가로 스크롤 방지) - '참고용' 스타일 기반 보완 */
  .var-table-wrap { width: 100%; max-width: 100%; overflow-x: visible !important; }
  .var-table {
    width: 100%;
    max-width: 100%;

    font-weight: 700;
  }
  .var-table th, .var-table td {

### Collections
게시글(<code class="code-inline">posts</code>)과 초안(<code class="code-inline">drafts</code>)를 포함한 컬렉션은 <code class="code-inline">_config.yml</code>의 컬렉션 설정에서 전역 permalink 규칙을 덮어쓸 수 있다.

```yaml
collections:
  my_collection:
    output: true
    permalink: /:collection/:name
```

# Themes
<br>
Jekyll은 커뮤니티가 유지·관리하는 템플릿과 스타일을 활용해 사이트를 커스터마이징할 수 있는 확장된 테마 시스템을 제공한다. Jekyll 테마는 사용할 플러그인을 지정하고, 정적 파일(assets), 레이아웃(layouts), 인클루드(includes), 스타일시트(stylesheets)를 사이트 파일로 재정의(override)할 수 있는 형태로 패키징하여 제공한다.

## Pick up a theme
아래 갤러리에서 사용할 테마를 찾아볼 수 있다.

- [GitHub #jekyll-theme 저장소](https://github.com/topics/jekyll-theme)
  
- [jamstackthemes.dev](https://jamstackthemes.dev/ssg/jekyll/)
  
- [jekyllthemes.org](https://jekyllthemes.org/)
  
- [jekyllthemes.io](https://jekyllthemes.io/)
  
- [jekyll-themes.com](https://jekyll-themes.com/)

다음 문서도 함께 참고하라: [resources](https://jekyllrb.com/resources/)

## Understanding gem-based themes
<code class="code-inline">jekyll new <PATH></code> 명령으로 새 Jekyll 사이트를 만들면, 기본으로 Minima라는 gem 기반 테마를 사용한다.

gem 기반 테마를 쓰면 사이트의 일부 디렉터리(예: <code class="code-inline">assets</code>, <code class="code-inline">_data</code>, <code class="code-inline">_layouts</code>, <code class="code-inline">_includes</code>, <code class="code-inline">_sass</code> 등)가 테마 gem 내부에 저장되어 프로젝트 루트에서 바로 보이지 않게 된다. Jekyll 빌드 과정에서 필요한 디렉터리는 모두 읽고 처리된다.

Minima를 쓸 경우 디렉터리 구조는 다음과 같다:

```
.
├── Gemfile
├── Gemfile.lock
├── _config.yml
├── _posts
│   └── 2016-12-04-welcome-to-jekyll.markdown
├── about.markdown
└── index.markdown
```

Gemfile과 Gemfile.lock은 Bundler가 사이트 빌드에 필요한 gem과 gem 버전을 기록·관리하는 데 사용된다.

gem 기반 테마를 쓰면, 개발자가 올린 업데이트가 테마 gem 사용자에게 쉽게 배포된다. 업데이트가 있으면 테마 개발자는 그 업데이트를 RubyGems에 배포한다.

theme gem이 설치되어 있다면 <code class="code-inline">bundle update</code>를 실행하여 프로젝트의 모든 gem을 업데이트할 수 있다. 테마 gem만 업데이트하려면 <code class="code-inline">bundle update <THEME></code>를 실행한다. <code class="code-inline"><THEME></code>에는 테마 이름(예: minima)을 넣는다. 테마 개발자가 추가하거나 변경한 파일(예: 스타일시트, includes 등)은 업데이트 시 자동으로 프로젝트에 반영된다.

gem 기반 테마의 목표는, 테마 파일이 디렉터리를 복잡하게 만들거나 콘텐츠를 만드는 메인 작업 흐름을 방해하지 않게 하면서도 꾸준히 업데이트되는 견고한 테마의 이점을 그대로 제공하는 것이다.

## Overriding theme defaults
Jekyll 테마는 기본 데이터, 레이아웃, include, 스타일시트를 가지고 있다. 하지만 자체 콘텐츠를 추가해 테마의 기본값을 재정의할 수 있다.

테마의 레이아웃이나 includes를 교체하려면, 변경하려는 파일의 <code class="code-inline">_layouts</code> 또는 <code class="code-inline">_includes</code> 디렉터리에 복사본을 만들거나, 덮어쓸 파일과 동일한 이름의 파일을 새롭게 생성한다.

예를 들어 선택한 테마에 <code class="code-inline">page</code> 레이아웃이 있다면, <code class="code-inline">_layouts</code> 디렉터리에 자체 <code class="code-inline">page</code>를 만들어 기존 테마의 레이아웃을 덮어쓸 수 있다(<code class="code-inline">_layouts/page.html</code>).

컴퓨터에 설치된 테마 파일 경로를 확인하는 법은 다음과 같다:

1. 테마 gem 이름을 <code class="code-inline">bundle info --path</code> 뒤에 붙여 실행한다(예: Jekyll 기본 테마의 경우 <code class="code-inline">bundle info --path minima</code>). 

이 명령은 gem 기반 테마 파일의 위치를 반환한다. 예를 들어, macOS에서 Minima 테마 파일의 예시 경로는 <code class="code-inline">/usr/local/lib/ruby/gems/2.6.0/gems/minima-2.5.1</code> 이다.

2. Finder(또는 Explorer)에서 해당 디렉터리를 연다:

```terminal
# macOS
open $(bundle info --path minima)

# Windows
# 먼저 gem 설치 경로를 확인:
#
#   bundle info --path minima
#   => C:/Ruby26-x64/lib/ruby/gems/3.4.1/gems/minima-2.5.1
#
# 그런 다음 위 경로에서 `/`를 `\`로 바꿔 Explorer를 실행
explorer C:\Ruby26-x64\lib\ruby\gems\3.4.1\gems\minima-2.5.1

# Linux
xdg-open $(bundle info --path minima)
```

Finder(또는 Explorer) 창이 열리면 테마 파일과 디렉터리가 보인다. Minima 테마 gem에는 다음과 같은 파일이 포함되어 있다:

```
.
├── LICENSE.txt
├── README.md
├── _includes
│   ├── disqus_comments.html
│   ├── footer.html
│   ├── google-analytics.html
│   ├── head.html
│   ├── header.html
│   ├── icon-github.html
│   ├── icon-github.svg
│   ├── icon-twitter.html
│   └── icon-twitter.svg
├── _layouts
│   ├── default.html
│   ├── home.html
│   ├── page.html
│   └── post.html
├── _sass
│   ├── minima
│   │   ├── _base.scss
│   │   ├── _layout.scss
│   │   └── _syntax-highlighting.scss
│   └── minima.scss
└── assets
    └── main.scss
```

테마 파일 구성을 파악했으면, Jekyll 사이트 디렉터리에 같은 이름의 파일을 만들어 어떤 테마 파일이든 재정의할 수 있다.

두 번째 예로, Minima의 푸터를 교체하려 한다고 가정해보자. Jekyll 사이트에 <code class="code-inline">_includes</code> 폴더를 만들고 <code class="code-inline">footer.html</code>라는 파일을 추가한다. 이제 Jekyll은 Minima 테마 gem의 <code class="code-inline">footer.html</code> 대신 사이트 자체 <code class="code-inline">footer.html</code>을 사용할 것이다.

스타일시트(stylesheet)를 수정하려면 한 단계 더 필요하다. 테마의 메인 Sass 파일(Minima의 경우 <code class="code-inline">_sass/minima.scss</code>)을 사이트 소스의 <code class="code-inline">_sass</code> 디렉터리로 복사해야 한다.

다음 폴더들에서 파일이 참조될 때 Jekyll은 항상 사이트의 파일을 먼저 확인하고, 없으면 테마의 기본 파일을 사용한다:

- <code class="code-inline">/assets</code>
  
- <code class="code-inline">/_data</code>
  
- <code class="code-inline">/_layouts</code>
  
- <code class="code-inline">/_includes</code>
  
- <code class="code-inline">/_sass</code>


테마 파일의 사본을 만들어 사용하는 경우, 사본에 대해서는 이후 테마 업데이트가 반영되지 않는다는 점에 유의하라. 테마의 모든 스타일시트 업데이트를 계속 반영하려면, 별도 CSS 파일을 추가로 만들어서 명시도(specificity)가 더 높은 CSS 선택자를 사용하여 필요한 부분만 덮어쓴다.

<div class="blue-caution" markdown="1">
ℹ️ 자세한 내용(어떤 파일을 재정의할 수 있는지 등)은 선택한 테마의 문서와 소스 리포지토리를 참고하라.
</div>

### Themes with `_data` directory <span class="ver-badge">4.3.0</span>
버전 4.3.0부터 Jekyll은 테마의 <code class="code-inline">_data</code> 디렉터리도 인식한다. 이로써 데이터(문구, 제목 등)를 테마에 함께 배포할 수 있다.

일반적인 사례로, 디자인 요소에서 사용하는 텍스트가 있다.

예를 들어, 어떤 테마가 <code class="code-inline">testimonials.html</code>이라는 include 파일을 제공한다고 하자. 이 디자인 요소는 페이지에 새로운 섹션을 만들고, 후기 목록 위에 h3 수준의 제목을 출력한다.

테마 개발자는 보통 이 제목을 영어로 작성해 HTML 소스에 직접 넣는다.

테마 사용자는 테마에 포함된 파일을 프로젝트로 복사해 그 안의 제목을 바꿀 수 있다.

하지만 <code class="code-inline">_data</code> 디렉터리를 활용하면, 이 작업을 다른 방식으로 처리할 수 있다.

테마 개발자가 디자인 템플릿에 텍스트를 하드코딩하는 대신 문자열 리소스(예: <code class="code-inline">site.data.i18n.testimonials.header</code>)를 참조하도록 하고, 테마의 데이터 디렉터리에 <code class="code-inline">_data/i18n/testimonials.yml</code> 파일을 생성한다.

이 파일에서 제목을 <code class="code-inline">header</code> 키에 지정해두면, 나머지는 Jekyll이 알아서 처리한다.

언뜻 보면 테마 개발자 입장에서는 이전보다 더 번거로운 방식으로 보일 수 있다.

그러나 테마 사용자 입장에서 커스터마이징이 훨씬 용이해진다.

예를 들어, 독일 사용자가 이 테마를 쓴다고 하자. 그 사용자가 후기 섹션의 제목을 독일어로 바꾸려면,프로젝트 디렉터리에 <code class="code-inline">site.data.i18n.testimonials.header</code> 키를 가진 데이터 파일을 하나 만들고, 키 값으로 독일어 번역(또는 원하는 제목)을 넣기만 하면 된다. 그러면 해당 섹션은 즉시 커스터마이징된다.

이제 include 파일을 프로젝트로 복사해 수정할 필요가 없고, 무엇보다 테마 업데이트를 지속적으로 받을 수 있다. 테마 개발자가 텍스트 모듈을 데이터 파일에서 중앙 관리·변경할 수 있도록 만들어 두었기 때문이다.

<div class="red-caution" markdown="1">
⚠️ 데이터 파일은 매우 높은 유연성을 가진다. 다만 테마 개발자가 텍스트 모듈을 배치한 위치와 테마 사용자가 배치하려는 위치가 서로 다를 경우 예기치 못한 오류가 발생할 수 있다.
</div>

위 예시와 관련해, 테마의 _data/i18n/testimonials.yml에 정의된 site.data.i18n.testimonials.header 키를 사용자 사이트에서 오버라이드할 경우, 키는 다음 세 가지 위치 중 한 곳에 정의할 수 있다.

- _data/i18n.yml에 testimonials.header 중첩 키로 정의.

- _data/i18n/testimonials.yml에 header로 정의(예시와 동일한 구조).

- _data/i18n/testimonials/header.yml에서 별도의 키 없이 파일 본문에 바로 제목 작성.

테마가 제공하는 디자인 요소의 텍스트 모듈을 어디에 어떤 키로 설정할지 사용자가 혼란스러워할 수 있으므로, 테마 개발자는 이 모호성을 염두에 두고 권장 경로와 키 구조를 명확히 안내해야 한다.

<div class="blue-caution" markdown="1">
ℹ️ 데이터 기능을 사용할 때는, 도입하려는 키가 테마의 동작 방식을 바꾸는지 아니면 화면에 출력되는 단순 데이터인지 구분해야 한다. 테마 동작에 영향을 준다면 <code class="code-inline">site.config</code>에 설정해야 하고, 단순 데이터라면 <code class="code-inline">site.data</code>로 제공해도 무방하다.
</div>

테마의 동작을 변경하는 데이터를 테마에 번들링해 배포하는 것은 **안티패턴(anti-pattern)**이기 때문에 사용을 권장하지 않는다. 테마 개발자는 사용자가 원할 경우 테마에서 제공하는 모든 데이터를 쉽게 오버라이드할 수 있도록 보장해야 한다.

## Converting gem-based themes to regular themes
gem 기반 테마를 제거하고 테마 gem에 아무 것도 남기지 않은 채, 모든 파일이 Jekyll 사이트 디렉터리에만 존재하는 일반 테마로 변환하려 한다고 가정하자.

이를 위해, 테마 gem의 디렉터리에서 파일들을 Jekyll 사이트 디렉터리로 복사한다. (예를 들어, 사이트를 <code class="code-inline">/myblog</code>에 만들었다면 해당 테마 파일들을 <code class="code-inline">/myblog</code>로 복사한다. 자세한 내용은 이전 섹션을 참고하라.)

그 다음, 테마에서 참조하던 플러그인들을 Jekyll에 알려줘야 한다. 이 플러그인들은 테마 gemspec 파일의 런타임 의존성(runtime dependencies) 항목에서 찾을 수 있다. 예를 들어 Minima 테마를 변환 중이라면 다음과 같다:

```
spec.add_runtime_dependency "jekyll-feed", "~> 0.12"
spec.add_runtime_dependency "jekyll-seo-tag", "~> 2.6"
```

이 의존성 항목들을 다음 두 가지 방식 중 하나로 <code class="code-inline">Gemfile</code>에 포함해야 한다.

먼저, <code class="code-inline">Gemfile</code>과 <code class="code-inline">_config.yml</code>에 각각 추가하는 방법이 있다.

```
# ./Gemfile
gem "jekyll-feed", "~> 0.12"
gem "jekyll-seo-tag", "~> 2.6"
```

```yaml
# ./_config.yml
plugins:
  - jekyll-feed
  - jekyll-seo-tag
```

또는 <code class="code-inline">_config.yml</code>은 변경하지 않고 Gemfile에서 Jekyll 플러그인으로 명시하는 방법도 있다. 

```
# ./Gemfile
group :jekyll_plugins do
  gem "jekyll-feed", "~> 0.12"
  gem "jekyll-seo-tag", "~> 2.6"
end
```

어느 쪽이든 <code class="code-inline">bundle update</code>를 실행하는 것을 잊지 말자.

깃허브 페이지에 배포하는 경우, 깃허브 페이지는 Bundler를 통해 플러그인을 불러오지 않으므로 <code class="code-inline">_config.yml</code>만 업데이트해야 한다.

마지막으로, <code class="code-inline">Gemfile</code>과 설정에서 테마 gem에 대한 참조를 제거한다. 예를 들어 <code class="code-inline">minima</code>를 제거하려면:

- <code class="code-inline">Gemfile</code>을 열어 <code class="code-inline">gem "minima", "~> 2.5"</code>를 삭제한다.

- <code class="code-inline">_config.yml</code>을 열어 <code class="code-inline">theme: minima</code>를 삭제한다.

이제 더 이상 <code class="code-inline">bundle update</code>를 실행해도 해당 테마 젬의 업데이트를 받아오지 않는다.

## Installing a gem-based theme
jekyll new <PATH> 명령 외에도 gem 기반 테마를 사용해 새로운 Jekyll 사이트를 만들 수 있다. 인터넷에서 gem 기반 테마를 찾아 Jekyll 프로젝트에 통합해 사용할 수 있다.

예를 들어 [RubyGems.org에서 jekyll theme](https://rubygems.org/search?utf8=%E2%9C%93&query=jekyll-theme)를 검색하면 여러 gem 기반 테마들을 찾을 수 있다. (모든 테마가 테마 이름에 관례적으로 <code class="code-inline">jekyll-theme를</code> 사용하지는 않는다.)

gem 기반 테마 설치 방법은 다음과 같다:

1. 사용할 테마 gem을 <code class="code-inline">Gemfile</code>에  추가한다:

```
# ./Gemfile

# 예시: 원하는 테마 gem을 선언
gem "jekyll-theme-minimal"
```

<code class="code-inline">jekyll new</code>로 시작했다면, 기존의 <code class="code-inline">gem "minima", "~> 2.0"</code>를 원하는 테마로 교체한다. 

예:

```
# ./Gemfile

- gem "minima", "~> 2.5"
+ gem "jekyll-theme-minimal"
```

2. 테마를 설치한다:

```terminal
bundle install
```

3. 테마를 활성화하기 위해 사이트의 <code class="code-inline">_config.yml</code>에 다음을 추가한다:

```yaml
theme: jekyll-theme-minimal
```

4. 사이트를 빌드한다:

```terminal
bundle exec jekyll serve
```

<div class="blue-caution" markdown="1">
ℹ️ <code class="code-inline">Gemfile</code>에는 여러 테마를 추가할 수 있지만, <code class="code-inline">_config.yml</code>에서는 오직 하나의 테마만 선택할 수 있다.
</div>

[깃허브 페이지](https://docs.github.com/ko/pages)에 Jekyll 사이트를 게시하는 경우, 깃허브 페이지는 모든 gem 기반 테마를 지원하지는 않는다는 점에 유의하라. 또한 깃허브 페이지에서는 <code class="code-inline">remote_theme</code> 설정을 사용해 [GitHub에 호스팅된 모든 Jekyll 테마](https://docs.github.com/ko/pages/setting-up-a-github-pages-site-with-jekyll/adding-a-theme-to-your-github-pages-site-using-jekyll#adding-a-jekyll-theme-in-your-sites-_configyml-file)를 gem 기반 테마처럼 사용할 수 있다.

## Creating a gem-based theme
Jekyll 테마 개발자가 테마를 gem으로 패키징해 RubyGems에 배포하면 사용자가 Bundler로 설치할 수 있다.

Ruby gem 제작이 처음이라도 걱정할 필요는 없다. <code class="code-inline">new-theme</code> 명령을 실행하면 Jekyll이 새 테마의 기본 골격을 만들어 준다. 테마 이름을 인자로 넘겨 <code class="code-inline">jekyll new-theme</code>를 실행한다.

예시는 다음과 같다:

```terminal
jekyll new-theme jekyll-theme-awesome
    create /path/to/jekyll-theme-awesome/_layouts
    create /path/to/jekyll-theme-awesome/_includes
    create /path/to/jekyll-theme-awesome/_sass
    create /path/to/jekyll-theme-awesome/_layouts/page.html
    create /path/to/jekyll-theme-awesome/_layouts/post.html
    create /path/to/jekyll-theme-awesome/_layouts/default.html
    create /path/to/jekyll-theme-awesome/Gemfile
    create /path/to/jekyll-theme-awesome/jekyll-theme-awesome.gemspec
    create /path/to/jekyll-theme-awesome/README.md
    create /path/to/jekyll-theme-awesome/LICENSE.txt
    initialize /path/to/jekyll-theme-awesome/.git
    create /path/to/jekyll-theme-awesome/.gitignore
새 Jekyll 테마 jekyll-theme-awesome이 /path/to/jekyll-theme-awesome에 준비됐다!
시작 안내는 /path/to/jekyll-theme-awesome/README.md를 참고하라.
```

폴더 구조에 맞춰 템플릿 파일을 추가한 뒤, 필요에 맞게 <code class="code-inline">.gemspec</code>과 README를 작성해 마무리한다.

### Layouts and includes
테마의 레이아웃과 include는 일반 Jekyll 사이트와 동일하게 동작한다. 레이아웃은 테마의 <code class="code-inline">/_layouts</code> 폴더에, include는 테마의 <code class="code-inline">/_includes</code> 폴더에 둔다.

예를 들어, 테마에 <code class="code-inline">/_layouts/page.html</code>이 있고 특정 페이지의 Front matter에 <code class="code-inline">layout: page</code>가 지정되어 있다면, Jekyll은 먼저 사이트의 <code class="code-inline">_layouts</code> 폴더에서 <code class="code-inline">page</code> 레이아웃을 찾고, 없을 경우 테마의 <code class="code-inline">page</code> 레이아웃을 사용한다.

### Assets
빌드할 때 테마의 /assets 폴더에 있는 모든 파일은 사용자의 사이트에 같은 상대 경로의 파일이 없으면 그대로 복사된다. 여기에는 SCSS, 이미지, 웹폰트 등 어떤 종류의 정적 파일이든 포함된다. 이 폴더의 파일들은 Jekyll 페이지 및 정적 파일과 동일하게 동작한다:

- 파일 상단에 [Front matter]({% link _apis/jekyll/03.md %}#front_matter)가 있으면 렌더링된다.

- 파일 상단에 Front matter가 없으면 최종 사이트로 그대로 복사된다.

이 구조 덕분에 테마 제작자는 테마에는 기본적으로 <code class="code-inline">/assets/styles.scss</code>를 제공하면서, 레이아웃에서는 <code class="code-inline">/assets/styles.css</code>를 참조하도록 설계할 수 있다.

<code class="code-inline">/assets</code> 안의 모든 파일은 Jekyll을 사용할 때와 마찬가지로 빌드된 사이트의 <code class="code-inline">/assets</code> 폴더에 출력된다.

### Stylesheets
테마의 스타일시트는 Jekyll 사이트를 작성할 때와 마찬가지로 테마의 <code class="code-inline">_sass</code> 폴더에 배치한다.

```
_sass
└── jekyll-theme-awesome.scss
```

테마의 스타일은 사용자의 스타일시트에서 @import 지시어로 포함할 수 있다.

```
@import "{{ site.theme }}";
```

### Theme-gem dependencies <span class="ver-badge">3.5.0</span>
Jekyll은 사이트의 설정 파일에서 <code class="code-inline">plugins</code> 배열에 명시하지 않아도, 테마 gem의 허용 목록에 포함된 <code class="code-inline">runtime_dependencies</code>를 자동으로 불러온다. (참고: 허용 목록은 <code class="code-inline">--safe</code> 옵션으로 빌드하거나 서버를 실행할 때에만 필요하다.)

이로써 최종 사용자는 테마 gem이 의도대로 동작하는 데 필요한 플러그인을 설정 파일에서 따로 관리할 필요가 없다.

### Pre-configuring Theme-gems <span class="ver-badge">4.0</span>
Jekyll은 테마 gem의 루트에 있는 _config.yml</code>을 읽어 들여, 그 내용을 사이트의 기존 설정 데이터와 병합한다.

다만 테마에서 로드되는 다른 항목들과 달리, 설정 파일을 로드할 때에는 아래와 같은 몇 가지 제약 사항이 있다:

- Jekyll 기본 설정은 테마의 설정 파일(theme-config)로 덮어쓸 수 없다. 이 권한은 사용자에게만 있다.

- 테마 설정 파일은 심볼릭 링크일 수 없다. 이는 safe mode 여부와 무관하며, 심볼릭 링크가 테마 gem의 정식 파일을 가리키더라도 허용되지 않는다.

- 테마 설정 파일은 키-값 쌍의 집합이어야 한다. 비어 있는 설정 파일, 특정 키 아래에 단순히 항목만 나열한 파일, 단순 문자열만 담긴 파일은 별도 경고나 로그 없이 무시된다.

- 테마 설정 파일에서 정의한 모든 설정은 사용자가 다시 덮어쓸 수 있다.

이 기능의 목적은 테마 도입을 더 쉽게 하는 것이지만, 위 제약을 통해 테마 설정 파일이 빌드에 바람직하지 않은 영향을 주지 않도록 보장한다. 테마가 필요로 하는 플러그인은 사용자가 설정 파일에 직접 기재하거나, 테마의 gemspec 파일에서 제공해야 한다.
이 기능의 목적은 테마 도입을 더 쉽게 하는 것이지만, 위 제약을 통해 테마 설정 파일이 빌드에 바람직하지 않은 영향을 주지 않도록 보장한다. 테마가 필요로 하는 플러그인은 사용자가 설정 파일에 직접 기재하거나, 테마의 <code class="code-inline">gemspec</code> 파일에서 제공해야 한다.

이 기능 덕분에 테마 gem은 테마 전용 설정 변수를 별도 설정 없이 바로 사용할 수 있다.

### Documenting your theme
테마에는 사이트 제작자가 테마를 설치하고 사용하는 방법을 설명하는 <code class="code-inline">README.md</code> 파일이 포함되어야 한다. 어떤 레이아웃(layouts)이 들어 있는지, 어떤 인클루드(includes)가 있는지, 사이트의 설정 파일에 특별히 추가해야 할 항목이 있는지 등을 안내하라.

### Adding a screenshot
테마는 시각적인 결과물이므로, 사용자가 테마의 모습을 바로 볼 수 있게 스크린샷(<code class="code-inline">screenshot.png</code>)을 테마 저장소의 루트에 포함해 두어라. 이렇게 하면 프로그램에서 자동으로 스크린샷을 가져올 수 있다. 필요하다면 이 스크린샷을 테마 문서에 포함하는 것도 가능하다.

### Previewing your theme
테마를 제작하면서 중간 결과물을 미리 보려면, <code class="code-inline">/index.html</code>나 <code class="code-inline">/page.html</code> 같은 파일에 더미 콘텐츠를 추가해 두는 것이 유용하다. 그러면 <code class="code-inline">jekyll build</code>와 <code class="code-inline">jekyll serve</code> 명령을 써서 일반 Jekyll 사이트를 미리 보듯이 테마를 미리 볼 수 있다.

<div class="blue-caution" markdown="1">
ℹ️ 테마를 로컬에서 미리 본다면, 배포 시 빌드된 사이트가 함께 포함되지 않도록 테마의 <code class="code-inline">.gitignore</code> 파일에 <code class="code-inline">/_site</code>를 반드시 추가하라.
</div>

### Publishing your theme
테마는 [RubyGems.org](https://rubygems.org/)를 통해 배포한다. 이를 위해 RubyGems 계정이 필요하며, [무료로 만들 수 있다](https://rubygems.org/sign_up).

1. 먼저, Git 저장소를 초기화한다:

```terminal
git init        # 처음 한 번만 실행
git add -A
git commit -m "Init commit"
```

2. 다음으로, 아래 명령을 실행해 테마를 패키징한다. <code class="code-inline">jekyll-theme-awesome</code>를 자신의 테마 이름으로 바꾼다:

```terminal
gem build jekyll-theme-awesome.gemspec
```

3. 마지막으로, 아래 명령을 실행해 패키징한 테마를 RubyGems에 업로드한다. 마찬가지로 <code class="code-inline">jekyll-theme-awesome</code>을 자신의 테마 이름으로 바꾼다.:

```terminal
gem push jekyll-theme-awesome-*.gem
```

4. 새 버전을 배포하려면, gemspec 파일(이 예에서는 jekyll-theme-awesome.gemspec)의 버전 번호를 올린 뒤 위의 1–3단계를 반복한다. 테마 버전을 올릴 때는 [Semantic Versioning](https://semver.org/)을 따르는 것을 권장한다.

# Pagination
<br>
블로그를 포함한 많은 웹사이트에서는 게시글 목록을 여러 페이지로 나눠서 보여준다. Jekyll은 페이지네이션(pagination) 플러그인을 제공하므로, 페이지별 목록에 필요한 파일과 폴더를 자동으로 생성할 수 있다.

Jekyll 3 이상 버전에서는 <code class="code-inline">jekyll-paginate</code> 플러그인을 Gemfile과 <code class="code-inline">_config.yml</code>의 <code class="code-inline">plugins</code> 항목에 추가한다. Jekyll 2에서도 이와 같은 방식이 표준이다.

<div class="blue-caution" markdown="1">
ℹ️ **페이지네이션은 HTML 파일에서만 동작한다.**

페이지네이션은 Jekyll 사이트의 Markdown 파일에서는 동작하지 않는다. 페이지네이션은 HTML 파일(<code class="code-inline">index.html</code>) 내부에서 호출할 때 작동한다. 이 파일은 하위 디렉터리에 둘 수 있으며, <code class="code-inline">paginate_path</code> 설정값에 따라 해당 하위 디렉터리 기준으로 페이징된 페이지들이 생성된다.
</div>

## Enable pagination
블로그 게시글에 페이지네이션을 적용하려면, 페이지당 표시할 글 개수를 <code class="code-inline">_config.yml</code>에 지정한다:

```yaml
paginate: 5
```

여기 지정한 숫자는 빌드된 사이트에서 페이지당 최대 게시물 수를 의미한다.

또한 페이징된 페이지들의 출력 경로를 지정할 수 있다:

```yaml
paginate_path: "/blog/page:num/"
```

이 설정은 blog/index.html을 읽고, 각 페이징된 페이지를 Liquid에 paginator 객체로 전달한 뒤 결과물을 blog/page:num/에 생성한다. 여기서 :num은 페이지 번호이며 2부터 시작한다. 만약 사이트에 12개의 게시글이 있고 paginate: 5로 설정한 경우, Jekyll은 첫 5개 게시글을 담은 blog/index.html, 다음 5개를 담은 blog/page2/index.html, 마지막 2개를 담은 blog/page3/index.html을 출력 디렉터리에 생성한다.

<div class="red-caution" markdown="1">
⚠️ **permalink를 지정하지 말 것.**

블로그 페이지의 Front matter에 permalink를 지정하면 페이지네이션이 깨지므로, permalink 항목은 제외한다.
</div>

<div class="blue-caution" markdown="1">
ℹ️ **카테고리, 태그, 컬렉션 페이지네이션.**

더 많은 기능이 필요하면 최신 플러그인 [jekyll-paginate-v2](https://github.com/sverrirs/jekyll-paginate-v2)를 사용할 수 있다. 사용법은 리포지토리의 [페이지네이션 예제](https://github.com/sverrirs/jekyll-paginate-v2/tree/master/examples)를 참고하라. **단, 이 플러그인은 깃허브 페이지에서 지원되지 않는다.**
</div>

<div class="blue-caution" markdown="1">
ℹ️ **페이지네이션은 태그나 카테고리를 지원하지 않는다.**

페이지네이션은 Front matter에 <code class="code-inline">hidden: true</code>가 지정된 게시글을 제외하고, <code class="code-inline">posts</code> 변수의 모든 게시글을 페이징 처리한다.
그러나 현재 공통 태그나 카테고리로 묶인 게시글 그룹은 대상으로 하는 페이징은 지원하지 않는다. 또한 페이지네이션은 게시글에만 적용되므로, 컬렉션 문서는 페이지네이션 대상에 포함되지 않는다.
</div>

## Render the paginated Posts
다음 단계는 paginator 변수를 이용해 게시글을 목록 형태로 출력하는 일이다. 보통 이 작업은 사이트의 주요 페이지 중 하나에서 하게 된다. 아래는 HTML 파일에서 페이지네이션된 게시글을 렌더링하는 간단한 예시다:

```liquid
---
layout: default
title: My Blog
---

<!-- 페이지네이션된 게시글을 순회 -->
{% raw %}{% for post in paginator.posts %}
  <h1><a href="{{ post.url }}">{{ post.title }}</a></h1>
  <p class="author">
    <span class="date">{{ post.date }}</span>
  </p>
  <div class="content">
    {{ post.content }}
  </div>
{% endfor %}

<!-- 페이지네이션 링크 -->
<div class="pagination">
  {% if paginator.previous_page %}
    <a href="{{ paginator.previous_page_path }}" class="previous">
      Previous
    </a>
  {% else %}
    <span class="previous">Previous</span>
  {% endif %}
  <span class="page_number ">
    Page: {{ paginator.page }} of {{ paginator.total_pages }}
  </span>
  {% if paginator.next_page %}
    <a href="{{ paginator.next_page_path }}" class="next">Next</a>
  {% else %}
    <span class="next ">Next</span>
  {% endif %}{% endraw %}
</div>
```

<div class="red-caution" markdown="1">
⚠️ **첫 페이지 예외(edge case) 주의**

Jekyll은 'page1' 폴더를 생성하지 않으므로, /page1 링크가 만들어지는 상황에서는 위 코드가 동작하지 않는다. 이 경우를 처리해야 한다면 아래 방법을 참고하라.
</div>

다음 HTML 코드는 첫 페이지를 처리하고, 현재 페이지를 제외한 나머지 페이지 번호에만 링크가 걸린 번호 목록을 렌더링한다.

```liquid
{% raw %}{% if paginator.total_pages > 1 %}
<div class="pagination">
  {% if paginator.previous_page %}
    <a href="{{ paginator.previous_page_path | relative_url }}">&laquo; Prev</a>
  {% else %}
    <span>&laquo; Prev</span>
  {% endif %}

  {% for page in (1..paginator.total_pages) %}
    {% if page == paginator.page %}
      <em>{{ page }}</em>
    {% elsif page == 1 %}
      <a href="{{ site.paginate_path | relative_url | replace: 'page:num/', '' }}">{{ page }}</a>
    {% else %}
      <a href="{{ site.paginate_path | relative_url | replace: ':num', page }}">{{ page }}</a>
    {% endif %}
  {% endfor %}

  {% if paginator.next_page %}
    <a href="{{ paginator.next_page_path | relative_url }}">Next &raquo;</a>
  {% else %}
    <span>Next &raquo;</span>
  {% endif %}
</div>
{% endif %}{% endraw %}
```
