# Pages
<br>
페이지는 콘텐츠의 가장 기본이 되는 구성 요소이다. 페이지는 날짜 기반이 아니고, 구성원 목록이나 레시피처럼 여러 항목을 묶는 컬렉션도 아닌 독립형 콘텐츠에 적합하다.

가장 간단한 페이지 추가 방법은 루트 디렉터리에 적절한 파일명의 HTML 파일을 두는 것이다. 또는 <code class="code-inline">.md</code> 확장자와 Front matter를 사용해 Markdown으로 작성하면 빌드 시 HTML로 변환된다. 예를 들어 홈페이지, 소개, 연락처 페이지로 구성된 사이트의  루트 디렉터리와 각 페이지에 대응되는 URL은 다음과 같을 수 있다:

```
.
├── about.md      # => http://example.com/about.html
├── index.html    # => http://example.com/
└── contact.html  # => http://example.com/contact.html
```

페이지가 많다면 하위 디렉터리로 정리할 수 있다. 프로젝트 소스에서 페이지를 묶기 위해 사용한 디렉터리 구조는 빌드 시 <code class="code-inline">_site</code>에 동일하게 생성된다. 다만, 특정 페이지의 Front matter에 permalink를 별도로 지정하면, <code class="code-inline">_site</code>의 경로는 이에 맞춰 변경된다.

```
.
├── about.md              # => http://example.com/about.html
├── documentation         # (페이지 묶음용 디렉터리)
│   └── doc1.md           # => http://example.com/documentation/doc1.html
├── design                # (페이지 묶음용 디렉터리)
│   └── draft.md          # => http://example.com/design/draft.html
```

## Changing the output URL
소스 파일의 폴더 구조와 출력 URL을 다르게 설정하고 싶다면 [permalinks]({% link _apis/jekyll/04.md %}#permalinks)를 이용하여 출력 URL을 자유롭게 지정할 수 있다. 

## Excerpts for pages <span class="ver-badge">4.1.1</span>
설정 파일에서 <code class="code-inline">page_excerpts: true</code>로 지정하면 페이지 요약(excerpt)를 자동으로 생성한다.

# Posts
<br>
Jekyll에는 블로그 기능이 기본으로 내장되어 있다. 블로그 글을 텍스트 파일로 작성하면 Jekyll이 블로그로 변환하는 데 필요한 모든 것을 제공한다.

## The Posts Folder
블로그 글은 <code class="code-inline">_posts</code> 폴더에 둔다. 보통 게시글은 [Markdown](https://daringfireball.net/projects/markdown/)으로 작성하며, HTML도 지원한다.

## Creating Posts
새 글을 만들려면 <code class="code-inline">_posts</code> 디렉터리에 다음 형식의 파일을 추가한다:

```
YEAR-MONTH-DAY-title.MARKUP
```

YEAR는 4자리, MONTH와 DAY는 2자리 숫자이며 MARKUP은 파일에서 사용하는 포맷을 나타내는 확장자이다. 즉 예시는 다음과 같다:

```
2011-12-31-new-years-eve-is-awesome.md
2012-09-12-how-to-write-a-blog.md
```

모든 블로그 글 파일은 [Front matter](#front-matter)로 시작해야 하며, 이는 [레이아웃]({% link _apis/jekyll/04.md %}#layouts)이나 기타 메타데이터를 지정하는 데 사용된다. 가장 단순하게는 Front matter를 비워 둘 수도 있다:

```markdown
---
layout: post
title:  "Welcome to Jekyll!"
---

# Welcome

**Hello world**, this is my first Jekyll blog post.

I hope you like it!
```

<div class="yellow-caution" markdown="1">
✨ **ProTip™: 다른 글에 링크하기**

다른 글을 링크할 때 <code class="code-inline">post_url</code> 태그를 사용하면 사이트의 permalink 스타일이 바뀌어도 URL이 깨지지 않는다.
</div>

<div class="blue-caution"  markdown="1">
ℹ️ **문자 인코딩(character sets) 주의**

일부 콘텐츠 처리기는 특정 문자를 더 보기 좋게 바꾸기도 한다. 예를 들어 Redcarpet의 <code class="code-inline">smart</code> 확장은 표준 ASCII 따옴표 문자를 곡선 형태의 유니코드 문자로 바꾼다. 브라우저가 바뀐 문자를 올바르게 표시하게 하려면 레이아웃의 <code class="code-inline">&lt;head&gt;</code> 안에 <code class="code-inline">&lt;meta charset="utf-8"&gt;</code>를 추가하여 charset 메타값을 지정하라.
</div>

## Including images and resources
글 본문과 함께 이미지, 다운로드 파일 등의 정적 파일을 포함해야 할 때가 있다. 일반적인 방법은 프로젝트 루트에 <code class="code-inline">assets</code> 같은 폴더를 만들고 그 안에 이미지, 파일 등의 리소스를 넣는 것이다. 이후 어떤 글에서든 사이트 루트를 기준 경로로 삼아 파일을 링크할 수 있다. 최적의 방법은 사이트의 (서브)도메인과 경로 설정에 따라 달라지지만, 아래에 Markdown으로 작성한 간단한 예시 몇 가지를 소개한다.

게시글에 이미지 포함하기:
```markdown
... which is shown in the screenshot below:
![My helpful screenshot](/assets/screenshot.jpg)
```

PDF 다운로드 링크:
```markdown
... you can [get the PDF](/assets/mydoc.pdf) directly.
```

## Displaying an index of posts
[Liquid](https://shopify.github.io/liquid/)와 태그를 이용하면 다른 페이지에 게시글 인덱스를 쉽게 만들 수 있다. 다음은 블로그 게시글로 가는 링크 목록을 만드는 간단한 예시이다:

```liquid
<ul>
  {% raw %}{% for post in site.posts %}
    <li>
      <a href="{{ post.url }}">{{ post.title }}</a>
    </li>
  {% endfor %}{% endraw %}
</ul>
```

게시글의 표시 위치 및 방식, 사이트 구조는 전적으로 자유롭게 구성할 수 있다. 더 알고 싶다면 Jekyll의 [템플릿 동작 방식]({% link _apis/jekyll/04.md %}#liquid)을 참고하라.

위 예시에서 <code class="code-inline">post</code> 변수는 <code class="code-inline">for</code> 루프 내부에만 있다는 점에 주목하라. 현재 렌더링 중인 페이지나 게시글의 <code class="code-inline">for</code> 루프 외부 변수에 접근하려면 <code class="code-inline">page</code> 변수를 사용하라.

## Tags and Categories
Jekyll은 블로그 게시글의 *태그*(tags) 와 *카테고리*(categories)를 기본 기능으로 정식 지원한다.

### Tags
태그는 글의 Front matter에서 지정한다. 단일 항목은 <code class="code-inline">tag</code>, 여러 항목은 <code class="code-inline">tags</code> 키를 쓴다.

Jekyll은 다수 항목들은 tags 키에 매핑하므로, 문자열에 공백이 포함되어 있으면 자동으로 분할한다. 예를 들어, <code class="code-inline">tag: classic hollywood</code>는 하나의 개체(<code class="code-inline">"classic hollywood"</code>)로 처리되지만, <code class="code-inline">tags: classic hollywood</code>는 각 항목들로 이루어진 배열(<code class="code-inline">["classic", "hollywood"]</code>)로 처리된다.

어떤 Front matter 키를 사용하든 Jekyll은 메타데이터를 내부적으로는 복수형 키에 저장하고, 이 값을 Liquid 템플릿에 노출한다.

현재 사이트에 등록된 모든 태그는 <code class="code-inline">site.tags</code> 를 통해 Liquid 템플릿에서 접근할 수 있다. 페이지에서 site.tags 를 순회하면 각 항목은 길이가 2인 배열이다. 첫 번째 항목은 태그 이름이며, 두 번째 항목은 해당 태그를 가진 게시글들의 배열이다.

```liquid
{% raw %}{% for tag in site.tags %}
  <h3>{{ tag[0] }}</h3>
  <ul>
    {% for post in tag[1] %}
      <li><a href="{{ post.url }}">{{ post.title }}</a></li>
    {% endfor %}
  </ul>
{% endfor %}{% endraw %}
```

### Categories
카테고리는 태그와 유사하게 동작한다.

- 카테고리는 Front matter에서 <code class="code-inline">category</code> 또는 <code class="code-inline">categories</code> 키로 지정할 수 있으며, 분할·배열 처리 로직은 태그와 같다.

- 사이트에 등록된 모든 카테고리는 <code class="code-inline">site.categories</code>로 Liquid에 노출되며, 위 태그 루프와 같은 방식으로 순회할 수 있다.

아래부터는 카테고리와 태그의 차이점이다.

태그와 달리 카테고리는 게시글의 파일 경로로도 지정할 수 있다. <code class="code-inline">_posts</code> 상위의 모든 디렉터리는 카테고리로 인식된다. 예를 들어 게시글이 <code class="code-inline">movies/horror/_posts/2019-05-21-bride-of-chucky.markdown</code> 경로에 있다면 <code class="code-inline">movies</code>와 horror</code>가 자동으로 해당 게시글의 카테고리로 등록된다.

게시글의 Front matter에 categories를 설정하면, 경로에서 파생된 기존 카테고리 목록에 없는 항목만 추가된다.

카테고리와 태그의 가장 대표적인 차이는, 게시글의 카테고리는 [생성된 URL 경로]({% link _apis/jekyll/04.md %}#global)에 포함될 수 있지만 태그는 포함될 수 없다는 점이다.

따라서 위 예시글의 URL은 각각 Front matter에 <code class="code-inline">category: classic hollywood</code>로 설정되었을 경우 <code class="code-inline">movies/horror/classic%20hollywood/2019/05/21/bride-of-chucky.html</code>, <code class="code-inline">categories: classic hollywood</code>일 경우 <code class="code-inline">movies/horror/classic/hollywood/2019/05/21/bride-of-chucky.html</code>가 된다.

## Post excerpts
게시글의 미리보기는 <code class="code-inline">excerpt</code> 변수로 접근할 수 있다. 기본값은 게시글의 첫 번째 문단이며, Front matter 또는 <code class="code-inline">_config.yml</code>에 <code class="code-inline">excerpt_separator</code> 변수를 설정해 커스터마이징할 수 있다.

```markdown
---
excerpt_separator: <!--more-->
---

Excerpt with multiple paragraphs

Here's another paragraph in the excerpt.
<!--more-->
Out-of-excerpt
```

다음은 미리보기를 제공하는 블로그 게시글 목록을 출력하는 예시이다:

```liquid
<ul>
  {% raw %}{% for post in site.posts %}
    <li>
      <a href="{{ post.url }}">{{ post.title }}</a>
      {{ post.excerpt }}
    </li>
  {% endfor %}{% endraw %}
</ul>
```

## Drafts
초안(draft)은 파일명에 날짜가 없는 게시글(post)로, 작업 중이어서 당장은 발행하지 않을 글이다. 초안 기능을 사용하려면 루트 디렉터리에 <code class="code-inline">_drafts</code> 폴더를 만들고 첫 번째 초안 파일을 추가한다:

```
.
├── _drafts
│   └── a-draft-post.md
...
```

초안을 포함한 사이트를 미리 보려면, <code class="code-inline">jekyll serve</code> 또는 <code class="code-inline">jekyll build</code>에 --drafts 플래그를 붙여 실행하라. 각 초안의 날짜는 해당 파일의 수정 시간으로 지정되며, 따라서 현재 편집한 초안이 가장 최신 글이 된다.

# Front Matter
<br>

[YAML](https://yaml.org/) Front matter 블록이 포함된 파일은 Jekyll에서 특수 파일로 처리된다. Front matter는 파일의 맨 앞에 와야 하며, 세 개의 대시(---)로 감싼 올바른 YAML 형식이어야 한다. 기본 예시는 다음과 같다:

```yaml
---
layout: post
title: Blogging Like a Hacker
---
```

이 세 개의 대시 선 사이에는 미리 정의된 변수(아래 참고)를 설정할 수도 있고, 사용자 정의 변수를 만들어 사용할 수도 있다. 이 변수들은 해당 파일의 아래쪽 본문, 그리고 해당 페이지나 포스트가 사용하는 레이아웃(layout)이나 인클루드(include) 어디에서든 Liquid 태그로 접근할 수 있다.

<div class="red-caution"  markdown="1">
⚠️ **UTF-8 문자 인코딩 주의**

UTF-8 인코딩을 사용할 때는 파일에 BOM(Byte Order Mark) 표식이 포함되지 않도록 하라. 포함되어 있으면 Jekyll에서 심각한 문제가 발생할 수 있다. 특히 [Windows에서 Jekyll]({% link _apis/jekyll/docs/01/jekyll_on_windows.md %})을 실행하는 경우 더 주의가 필요하다.
</div>

<div class="yellow-caution" markdown="1">
✨ **Front Matter 변수는 선택 사항이다**

Liquid 태그와 변수를 쓰고 싶은데 Front matter 설정이 필요하지 않다면 비워 둬도 된다. 내용이 비어 있는 --- 구분선만 있어도 Jekyll은 해당 파일을 처리할 수 있다. (CSS나 RSS 피드 같은 파일에 유용하다.)
</div>

# Collections
<br>
컬렉션(collections)은 팀 구성원이나 콘퍼런스 발표처럼 관련된 콘텐츠를 묶어 관리하기에 적합하다.

## Setup
컬렉션을 사용하려면 먼저 <code class="code-inline">_config.yml</code>에 컬렉션을 정의해야 한다. 다음은 직원 컬렉션의 예시이다:

```yaml
collections:
  - staff_members
```

위 예시에서 <code class="code-inline">collections</code>는 각 컬렉션에 대한 추가 메타데이터 없이 시퀀스(배열)로 정의되어 있다. 컬렉션에 메타데이터를 지정하려면 시퀀스 대신 매핑(해시맵)으로 <code class="code-inline">collections</code>를 정의하고, 그 안에 다음과 같이 필드를 추가로 정의할 수 있다:

```yaml
collections:
  staff_members:
    people: true
```

<div class="blue-caution"  markdown="1">
ℹ️ 컬렉션을 시퀀스로 정의하면 해당 컬렉션의 페이지는 자동으로 렌더링되지 않는다. 이를 활성화하려면 컬렉션에 <code class="code-inline">output: true</code>를 지정해야 하며, 이 설정을 위해서는 컬렉션을 매핑 형태로 정의해야 한다. 자세한 내용은 [출력(Output)](#output) 섹션을 참고하라.
</div>

<div class="yellow-caution" markdown="1">
✨ **컬렉션 한곳에 모으기** <span class="ver-badge">3.7.0</span>

모든 컬렉션을 한 디렉터리에 모아두고 싶다면 <code class="code-inline">collections_dir: my_collections</code>처럼 디렉터리를 지정할 수 있다.
이제 Jekyll은 <code class="code-inline">books</code> 컬렉션을 <code class="code-inline">my_collections/_books</code>에서, <code class="code-inline">recipes</code> 컬렉션을 <code class="code-inline">my_collections/_recipes</code>에서 찾는다.
</div>

<div class="red-caution"  markdown="1">
⚠️ **초안과 게시글을 사용자 지정 컬렉션 디렉터리로 옮길 것**

<code class="code-inline">collections_dir: my_collections</code>로 모든 컬렉션을 한곳에 보관하도록 지정했다면, <code class="code-inline">_drafts</code>와 <code class="code-inline">_posts</code> 디렉터리도 각각 <code class="code-inline">my_collections/_drafts</code>, <code class="code-inline">my_collections/_posts</code>로 옮겨야 한다. 컬렉션을 담는 디렉터리 이름은 밑줄(_)로 시작할 수 없다는 점에 유의하라.
</div>

## Add content
해당 컬렉션에 대응하는 폴더(예: <code class="code-inline">&lt;source&gt;/_staff_members</code>)를 만들고 문서를 추가하라. Front matter가 있으면 Front matter를 먼저 처리하고, 그 이후의 모든 내용은 문서의 <code class="code-inline">content</code> 속성으로 들어간다. Front matter가 없으면 Jekyll은 해당 파일을 정적 파일로 간주하여 추가로 처리하지 않는다. Front matter가 있으면 Jekyll이 파일 내용을 처리해 설정에 맞는 최종 출력물로 변환한다.

Front matter의 존재 여부와 무관하게, 해당 컬렉션의 메타데이터에 output: true가 설정되어 있어야만 Jekyll이 출력 디렉터리(예: _site)에 최종 출력 파일을 생성한다.

아래는 위에서 설정한 컬렉션에 직원을 추가하는 예시이다. 파일명은 <code class="code-inline">./_staff_members/jane.md</code>이고, 내용은 다음과 같다:

```markdown
---
name: Jane Doe
position: Developer
---
Jane has worked on Jekyll for the past *five years*.
```

*[게시글(posts)](#posts)도 내부적으로는 컬렉션으로 취급되지만, 게시글에는 위 규칙이 적용되지 않는다는 점에 주목하라. 올바른 파일명 형식을 갖춘 게시글은 Front matter가 없어도 처리된다.*

<div class="blue-caution" markdown="1">
ℹ️ **디렉터리 이름을 정확히 지정할 것**
폴더 이름은 <code class="code-inline">_config.yml</code>에 정의한 컬렉션 이름과 동일해야 하며, 앞에 밑줄(_)을 붙여야 한다.
</div>

## Output
이제 페이지에서 <code class="code-inline">site.staff_members</code>를 순회하여 각 스태프 멤버에 해당하는 내용들을 출력할 수 있다. 게시글과 마찬가지로, 문서의 본문은 <code class="code-inline">content</code> 변수를 통해 접근한다:

```liquid
{% raw %}{% for staff_member in site.staff_members %}
  <h2>{{ staff_member.name }} - {{ staff_member.position }}</h2>
  <p>{{ staff_member.content | markdownify }}</p>
{% endfor %}{% endraw %}
```

Jekyll이 컬렉션의 각 문서에 대해 렌더링된 페이지를 생성하도록 하려면, <code class="code-inline">_config.yml</code>에서 컬렉션 메타데이터의 <code class="code-inline">output</code> 키를 <code class="code-inline">true</code>로 설정한다:

```yaml
collections:
  staff_members:
    output: true
```

<code class="code-inline">url</code> 속성을 이용하면 생성된 페이지에 링크를 걸 수 있다:

```liquid
{% raw %}{% for staff_member in site.staff_members %}
  <h2>
    <a href="{{ staff_member.url }}">
      {{ staff_member.name }} - {{ staff_member.position }}
    </a>
  </h2>
  <p>{{ staff_member.content | markdownify }}</p>
{% endfor %}{% endraw %}
```

## Permalinks
컬렉션 전체의 출력 URL을 제어하려면 [컬렉션 전용 permalink 변수들](#collections)을 이용할 수 있다.

## Custom Sorting of Documents <span class="ver-badge">4.0</span>
하나의 컬렉션에 두 개의 문서가 있을 때, 두 문서 모두 Front matter에 <code class="code-inline">date</code> 키가 있으면 기본적으로 두 문서는 <code class="code-inline">date</code> 속성을 기준으로 정렬된다. 반대로 하나 또는 두 문서 모두 date 키가 없으면, 각 문서의 경로를 기준으로 정렬된다.

이 정렬 방식은 컬렉션의 메타데이터로 제어할 수 있다.

### Sort By Front Matter Key
컬렉션의 메타데이터의 <code class="code-inline">sort_by</code>에 Front matter 키 이름(문자열)을 지정하면, 그 키를 기준으로 문서를 정렬할 수 있다. 예를 들어, 튜토리얼 컬렉션을 <code class="code-inline">lesson</code> 키 기준으로 정렬하려면 다음과 같이 설정한다:

```yaml
collections:
  tutorials:
    sort_by: lesson
```

문서는 해당 키 값을 기준으로 오름차순으로 배치된다. 특정 문서에 해당 Front matter 키가 없으면, 그 문서는 정렬된 문서들 바로 뒤에 놓인다. 키가 없는 문서가 여러 개인 경우, 이 문서들은 날짜나 경로 기준으로 정렬되고, 키 기준으로 정렬된 문서들 뒤에 배치된다.

### Manually Ordering Documents
원하는 순서를 파일명으로 직접 지정하려면, 컬렉션 메타데이터의 <code class="code-inline">order</code>에 **파일명을 원하는 순서대로 나열한다**. 예를 들어, 튜토리얼 컬렉션을 수동으로 정렬하는 법은 다음과 같다:

```yaml
collections:
  tutorials:
    order:
      - hello-world.md
      - introduction.md
      - basic-concepts.md
      - advanced-concepts.md
```

목록에 이름이 없는 파일은 재배치된 문서들 뒤에 자동으로 배치된다. 문서가 하위 디렉터리에 있다면, 항목에 경로까지 포함해야 한다:

```yaml
collections:
  tutorials:
    order:
      - hello-world.md
      - introduction.md
      - concepts/basics.md
      - concepts/advanced.md
```

<code class="code-inline">sort_by</code>와 <code class="code-inline">order</code>가 모두 정의된 경우, <code class="code-inline">order</code> 목록이 우선순위를 갖는다.

# Data Files
<br>
Jekyll이 제공하는 [내장 변수]({% link _apis/jekyll/04.md %}#variables) 외에도, [Liquid 템플릿 시스템](https://github.com/Shopify/liquid/wiki/Liquid-for-Designers)을 통해 접근할 수 있는 사용자 정의 데이터를 직접 지정할 수 있다.

Jekyll은 <code class="code-inline">_data</code> 디렉터리에 위치한 [YAML](https://yaml.org/), [JSON](https://www.json.org/json-en.html), [CSV](https://en.wikipedia.org/wiki/Comma-separated_values), [TSV](https://en.wikipedia.org/wiki/Tab-separated_values) 파일에서 데이터를 불러올 수 있다. 단, CSV와 TSV 파일에는 반드시 헤더 행이 있어야 한다.

이 기능을 사용하면 템플릿에서 같은 내용을 반복해서 쓰지 않아도 되고, <code class="code-inline">_config.yml</code>을 변경하지 않고도 사이트별 옵션을 설정할 수 있다.

플러그인과 테마도 데이터 파일을 활용해 설정값을 지정할 수 있다.

## The Data Folder
<code class="code-inline">_data</code> 폴더는 사이트 생성 시 Jekyll이 사용할 추가 데이터를 저장하는 곳이다. 이 폴더의 파일 형식은 YAML, JSON, TSV, 또는 CSV이어야 하며(확장자는 각각 <code class="code-inline">.yml</code>, <code class="code-inline">.yaml</code>, <code class="code-inline">.json</code>, <code class="code-inline">.tsv</code>, <code class="code-inline">.csv</code> 중 하나), <code class="code-inline">site.data</code>를 통해 접근할 수 있다.

## Example: List of members
다음은 데이터 파일을 사용하는 기본 예시로, Jekyll 템플릿에서 대량의 코드를 복사-붙여넣기하는 수고를 피할 수 있다:

<code class="code-inline">_data/members.yml</code>에 다음과 같이 작성한다:

```yaml
- name: Eric Mill
  github: konklone

- name: Parker Moore
  github: parkr

- name: Liu Fengyun
  github: liufengyun
```

<code class="code-inline">_data/members.csv</code>에는 다음과 같이 작성한다:

```
name,github
Eric Mill,konklone
Parker Moore,parkr
Liu Fengyun,liufengyun
```

이 데이터는 <code class="code-inline">site.data.members</code>로 접근할 수 있다. (파일의 확장자를 제외한 파일 이름(*basename*)이 변수명을 결정하므로, 같은 디렉터리에 확장자만 다른 동일한 basename의 데이터 파일을 함께 두지 않도록 주의하라.)

이제 템플릿에서 구성원 목록을 렌더링할 수 있다:

```liquid
<ul>
{% raw %}{% for member in site.data.members %}
  <li>
    <a href="https://github.com/{{ member.github }}">
      {{ member.name }}
    </a>
  </li>
{% endfor %}{% endraw %}
</ul>
```

## Subfolders
<code class="code-inline">_data</code> 폴더의 하위 폴더에도 데이터 파일을 둘 수 있다. 하위 폴더의 각 단계는 변수의 네임스페이스에 추가된다. 아래 예시는 GitHub 조직을 <code class="code-inline">orgs</code> 폴더 아래의 개별 파일로 나눠 정의하는 방법을 보여준다.

<code class="code-inline">_data/orgs/jekyll.yml</code>:

```yaml
username: jekyll
name: Jekyll
members:
  - name: Tom Preston-Werner
    github: mojombo

  - name: Parker Moore
    github: parkr
```

<code class="code-inline">_data/orgs/doeorg.yml</code>:

```yaml
username: doeorg
name: Doe Org
members:
  - name: John Doe
    github: jdoe
```

이렇게 정의한 조직 데이터에는 site.data.orgs 뒤에 파일명을 붙여 접근할 수 있다:

```liquid
<ul>
{% raw %}{% for org_hash in site.data.orgs %}
{% assign org = org_hash[1] %}
  <li>
    <a href="https://github.com/{{ org.username }}">
      {{ org.name }}
    </a>
    ({{ org.members | size }} members)
  </li>
{% endfor %}{% endraw %}
</ul>
```

## Example: Accessing a specific author
페이지와 게시글에서도 특정 데이터 항목에 접근할 수 있다. 아래 예시는 특정 항목에 접근하는 방법을 보여준다.

<code class="code-inline">_data/people.yml</code>:

```yaml
dave:
    name: David Smith
    twitter: DavidSilvaSmith
```

저자(author)는 글의 Front matter에서 페이지 변수로 지정한다:

```liquid
---
title: sample post
author: dave
---

{% raw %}{% assign author = site.data.people[page.author] %}
<a rel="author"
  href="https://twitter.com/{{ author.twitter }}"
  title="{{ author.name }}">
    {{ author.name }}{% endraw %}
</a>
```

사이트에 견고한 내비게이션 구조를 만드는 방법(특히 문서 사이트처럼 페이지가 많아 체계적인 구성이 필요한 경우)은 [Navigation]({% link _apis/jekyll/tutorials.md %})을 참고한다.

## CSV/TSV Parse Options
Ruby가 CSV와 TSV 파일을 파싱하는 방식은 <code class="code-inline">csv_reader</code>와 <code class="code-inline">tsv_reader</code> 설정 옵션으로 커스터마이징할 수 있다. 각 설정 키에서 제공하는 옵션은 동일하다.

- <code class="code-inline">converters</code>: 파일을 파싱할 때 사용할 [CSV 컨버터](https://ruby-doc.org/stdlib-2.5.0/libdoc/csv/rdoc/CSV.html#Converters)를 지정한다. 가능한 옵션은 <code class="code-inline">integer</code>, <code class="code-inline">float</code>, <code class="code-inline">numeric</code>, <code class="code-inline">date</code>, <code class="code-inline">date_time</code>, <code class="code-inline">all</code>이며, 기본값으로 이 목록은 비어 있다.

- <code class="code-inline">encoding</code>: 파일의 인코딩을 지정한다. 기본값은 사이트의 <code class="code-inline">encoding</code> 설정 값이다.

- <code class="code-inline">headers</code>: 첫 줄을 헤더 행으로 파싱할지 여부를 결정하는 Boolean 필드이다. <code class="code-inline">false</code>이면 첫 행도 데이터로 취급하며, 기본값은 <code class="code-inline">true</code>이다.

예시:

```yaml
csv_reader:
    converters:
      - numeric
      - datetime
    headers: true
    encoding: utf-8

tsv_reader:
    converters:
      - all
    headers: false
```

# Assets
<br>
Jekyll은 [Sass](https://sass-lang.com/)를 기본으로 지원하며, [CoffeeScript](https://coffeescript.org/)는 Ruby gem으로 사용할 수 있다. 이를 위해 먼저 올바른 확장자(<code class="code-inline">.sass</code>, <code class="code-inline">.scss</code>, <code class="code-inline">.coffee</code>)를 가진 파일을 만들고, **파일의 맨 앞을 다음과 같이 세 개의 대시(---)선으로 감싼 Front matter로 시작해야 한다**:

```
---
---

// start content
.my-definition
  font-size: 1.2em
```

Jekyll은 이러한 파일을 일반 페이지와 동일하게 처리하므로, 출력 파일은 소스 파일과 동일한 디렉터리에 생성된다. 예를 들어 사이트의 소스 폴더에 <code class="code-inline">css/styles.scss</code> 파일이 있으면, Jekyll은 이를 처리한 후 출력 폴더 하위의 <code class="code-inline">css/styles.css</code>로 생성한다.

<div class="blue-caution"  markdown="1">
ℹ️ Jekyll은 정적 파일 안의 Liquid 필터와 태그도 모두 처리한다.
[Mustache](https://mustache.github.io/)처럼 [Liquid 템플릿 문법]({% link _apis/jekyll/04.md %}#liquid)과 충돌하는 JavaScript 템플릿 언어를 사용하는 경우, 코드 주변을 {% raw %}와 {% endraw %}로 감싸야 한다.
</div>

## Sass/SCSS
Jekyll에서는 Sass 변환 동작을 몇 가지 방식으로 커스터마이징할 수 있다.

모든 파셜 파일은 <code class="code-inline">sass_dir</code>에 두며, <code class="code-inline">sass_dir</code>의 기본 경로는 <code class="code-inline">&lt;source&gt;/_sass</code>이다. 메인 SCSS나 Sass 파일은 최종 출력물이 생성될 디렉터리(예: <code class="code-inline">&lt;source&gt;/css</code>)에 둔다. 예시로 [Jekyll의 Sass 컴파일 기능을 사용하는 예제 사이트](https://github.com/jekyll/jekyll-sass-converter/tree/master/docs)를 참고하라.

Sass의 <code class="code-inline">@import</code> 구문을 사용하는 경우, <code class="code-inline">sass_dir</code>이 Sass 파일들이 위치한 기준 경로로 설정되어 있는지 확인하라:

```yaml
sass:
  sass_dir: _sass
```

Sass 변환기의 <code class="code-inline">sass_dir</code> 기본값은 <code class="code-inline">_sass</code>이다.

<div class="blue-caution"  markdown="1">
ℹ️ **<code class="code-inline">sass_dir</code>는 Sass에서만 사용된다.**

<code class="code-inline">sass_dir</code>는 Sass 임포트의 검색 경로(load path)로만 쓰인다. 따라서 Jekyll은 이 디렉터리의 파일을 직접 인식하지 않으며, 이 경로에 있는 파일에는 위에서 설명한 빈 Front matter가 있으면 안 된다. 빈 Front matter가 있더라도 앞서 설명한 방식대로 변환되지 않는다. 이 폴더에는 import 대상 파일만 두어야 한다.
</div>

또한 <code class="code-inline">_config.yml</code>의 <code class="code-inline">style</code> 옵션을 설정하여 출력 스타일을 지정할 수도 있다:

```yaml
sass:
  style: compressed
```

이 값들은 Sass로 그대로 전달되므로, Sass가 지원하는 모든 출력 스타일 옵션을 사용할 수 있다.

Sass 설정 옵션에 대한 자세한 내용은 [Sass 설정]({% link _apis/jekyll/docs/02/sass_scss_options.md %}) 문서를 참고하라.

## Coffeescript
Jekyll 3.0 이상에서 CoffeeScript를 사용하려면 다음을 수행한다.

- <code class="code-inline">jekyll-coffeescript</code> gem을 설치한다.

- <code class="code-inline">_config.yml</code>을 최신 상태로 업데이트하고 다음 내용을 추가한다:

```yaml
plugins:
  - jekyll-coffeescript
```

## Add front matter to static files
정적 파일에는 Front matter 값을 직접 추가할 수 없지만, 설정 파일의 [defaults 속성]({% link _apis/jekyll/docs/02/front_matter_defaults.md %})을 통해 Front matter 값을 지정할 수 있다. Jekyll이 사이트를 빌드할 때, 지정한 Front matter 값을 사용한다.

다음 예시를 참고하라:

<code class="code-inline">_config.yml</code> 파일의 <code class="code-inline">defaults</code> 속성에 다음 값을 추가한다:

```yaml
defaults:
  - scope:
      path: "assets/img"
    values:
      image: true
```

위 설정은 Jekyll 사이트에 이미지(정적 파일)를 저장한 <code class="code-inline">assets/img</code> 폴더 경로가 있다고 가정한다. 따라서 사이트를 빌드할 때 Jekyll은 각 이미지를 <code class="code-inline">image: true</code>라는 Front matter 값을 가진 것처럼 취급한다.

<code class="code-inline">assets/img</code>에 있는 모든 이미지 파일을 나열하고 싶다면, 다음과 같이 for 루프를 사용해 <code class="code-inline">site.static_files</code> 객체에서 해당 Front matter 속성을 가진 정적 파일만 가져올 수 있다:

```liquid
{% raw %}{% assign image_files = site.static_files | where: "image", true %}
{% for myimage in image_files %}
  {{ myimage.path }}
{% endfor %}{% endraw %}
```

사이트를 빌드하면, 위 Front matter 조건을 만족하는 각 파일의 경로가 목록으로 출력된다.
